# BUG-EDITOR-002: GET /internal/presence/:id requires no authentication (unauthenticated access)

**Date:** 2026-05-14
**Severity:** HIGH
**Environment:** dev (dev-ws.doable.me)
**Test Case:** TC-WS-AUTH-037

## Summary

`GET /internal/presence/:projectId` on the WebSocket server returns 200 with presence data for any caller with no authentication required. No `X-Internal-Secret` header or JWT is checked.

## Repro Steps

```
# No auth header - returns 200 with presence data
curl https://dev-ws.doable.me/internal/presence/a9bcb1a9-20ea-4ad5-a4e3-1ed9662284ac

# Response: 200 {"users":[...]}
```

## Expected

The `/internal/presence/:id` endpoint is an internal API intended for service-to-service calls. It should require the `X-Internal-Secret` header (same as `/internal/broadcast`) and return 403 for unauthenticated callers.

## Actual

200 response with presence data returned to any unauthenticated caller over the public Cloudflare tunnel.

## Impact

Any external attacker can enumerate which users are currently active in any project room by probing `/internal/presence/:projectId` with any known or guessed project UUID. This leaks:
- User IDs of active collaborators
- Display names
- Current file being edited
- Join timestamps and activity status

## Note

`GET /internal/collab-active` may have the same issue (not verified in this test run).
