# BUG-2026-05-14-publish-005: Login rate limiter Retry-After header not exposed in CORS headers

**Severity:** Low  
**Environment:** dev (https://dev.doable.me)  
**Date:** 2026-05-14  
**Affected TCs:** N/A (auth rate limiting)

## Summary
When the login rate limit is hit (10 attempts per 15 minutes), the server code sets a `Retry-After` header but it is not accessible cross-origin by browser clients. The header is also not included in CORS `Access-Control-Expose-Headers`, so browser fetch calls cannot read it.

## Steps to Reproduce
1. Make 10+ rapid POST requests to /auth/login
2. On the 429 response, check for `Retry-After` header

## Expected
- 429 response with `Retry-After: 900` (or similar seconds)
- Header accessible cross-origin via CORS expose headers
- UI shows "Please wait X minutes before trying again" with countdown

## Actual
- 429 response with `{"error":"Too many requests, please try again later."}`
- No `Retry-After` header visible in browser (even though server code sets it)
- No indication to user of when they can retry

## Code Reference
`services/api/src/middleware/rate-limit.ts` line 87: `c.header("Retry-After", String(resetSeconds));` — this is set but not included in CORS expose headers config.

## Impact
Users who hit the rate limit see a generic error with no indication of when they can retry. UX is poor — users may keep clicking and triggering more rate limits.
