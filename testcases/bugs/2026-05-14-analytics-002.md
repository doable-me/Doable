# BUG-2026-05-14-analytics-002: Puppeteer launched with --no-sandbox unconditionally

## Severity
P0 — Security: Chromium sandbox disabled for all environments regardless of user context

## Test Case
TC-THUMB-GEN-027

## Environment
- File: services/api/src/thumbnails/capture.ts
- Date: 2026-05-14

## Steps to Reproduce
1. Read `services/api/src/thumbnails/capture.ts` lines 38–45
2. Observe `puppeteer.launch()` args array

## Expected
Per TC-THUMB-GEN-027: `--no-sandbox` should only be passed when running as root (uid=0), which is unavoidable in some container setups. In all other cases the full Chromium sandbox should be active.

## Actual
```typescript
args: ["--no-sandbox", "--disable-setuid-sandbox", "--disable-gpu"],
```
These flags are hardcoded unconditionally. There is no check for `process.getuid?.() === 0` or any environment flag (e.g. `DOABLE_HARDENING`) to gate sandbox disabling.

## Impact
- Chromium sandbox provides critical isolation preventing renderer process escapes. Disabling it means a malicious published site (containing exploit JS) could potentially escape the renderer and access the host filesystem or network.
- TC-THUMB-GEN-030 (network egress restriction) and TC-THUMB-GEN-044 (DOABLE_HARDENING=full) are also unverifiable/ineffective without sandbox.
- Per CLAUDE.md: "Run with --no-sandbox only if root user (avoided); otherwise full sandbox; isolated user"

## Fix Direction
Make sandbox conditional:
```typescript
const isRoot = process.getuid?.() === 0;
const args = [
  ...(isRoot ? ["--no-sandbox", "--disable-setuid-sandbox"] : []),
  "--disable-gpu",
];
```
Better: run Puppeteer as a dedicated non-root user so `--no-sandbox` is never needed.
