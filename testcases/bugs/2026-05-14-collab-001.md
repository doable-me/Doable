# BUG-COLLAB-001: Login rate limiter blocks legitimate users too aggressively

**Date**: 2026-05-14
**Environment**: dev (dev.doable.me / dev-api.doable.me)
**Severity**: HIGH
**Area**: Authentication / Rate Limiting

## Summary
The login endpoint (`POST /auth/login`) applies an IP-based rate limit that triggers after approximately 5 attempts and enforces a lockout window longer than 60 seconds. During automated testing with two browser sessions (Chrome + Playwright) from the same machine, both sessions were blocked with HTTP 429 "Too many requests, please try again later." for extended periods.

## Steps to Reproduce
1. Attempt to log in via email/password on dev.doable.me/login
2. Fail login 3-5 times (wrong password or server error)
3. Attempt again with correct credentials
4. Observe "Too many requests, please try again later." error

## Expected
Rate limit should allow reasonable retry attempts (e.g., 10 per 15 minutes) and clearly communicate retry-after time to the user.

## Actual
- HTTP 429 returned after ~5 attempts
- Error message "Too many requests, please try again later." gives no retry-after time
- Lockout persists for >60 seconds
- Multiple legitimate users on same IP (office/NAT scenario) are all blocked simultaneously

## Evidence
- Console errors: 12+ consecutive `Failed to load resource: 429` on `/auth/login`
- Both Chrome and Playwright sessions blocked simultaneously from same IP
- Rate limit persists across 10s, 20s, 30s wait intervals

## Impact
Any scenario with multiple users behind the same NAT (common in offices or dev environments) will trigger the rate limit after a few failed attempts, blocking all users on that IP.
