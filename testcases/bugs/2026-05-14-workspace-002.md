# BUG-2026-05-14-WS-002: POST /projects without workspaceId silently uses caller's default workspace

**Date:** 2026-05-14
**Environment:** dev (dev-api.doable.me)
**Severity:** Medium
**Category:** Validation / Data Routing
**Test Case:** TC-PROJ-CREATE-008b

## Summary
`POST /projects` with `workspaceId` omitted returns `201` and silently assigns the project to the caller's first/default workspace instead of returning a validation error.

## Steps to Reproduce
1. Authenticate as `owner-pro@doable.me` (has multiple workspaces)
2. `POST /projects` body: `{"name":"NoWSTest"}` — no `workspaceId`
3. Observe: `201` response with `workspace_id: "492bdda5-..."` (caller's default workspace)

## Expected
`400 {"error":"Validation failed","details":{"workspaceId":["Required"]}}`

## Actual
`201` — project created in caller's default workspace without any indication

## Impact
- Multi-workspace users may accidentally create projects in the wrong workspace
- Clients that forget `workspaceId` get silent data routing to an unexpected workspace
- Spec states `workspaceId` is required

## Recommended Fix
Mark `workspaceId` as required in the Zod schema for `POST /projects`, or at minimum return a warning header when falling back to default workspace.
