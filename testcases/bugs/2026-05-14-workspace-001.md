# BUG-2026-05-14-WS-001: Invite accept does not verify email match — any user can accept any invite

**Date:** 2026-05-14
**Environment:** dev (dev-api.doable.me)
**Severity:** High
**Category:** Security / Access Control
**Test Case:** TC-WS-INV-028

## Summary
The `/workspaces/invite/accept` endpoint does not verify that the authenticated user's email matches the email address the invite was sent to. Any authenticated user who obtains an invite token can accept it and join the workspace with the invited role — regardless of whether the invite was meant for them.

## Steps to Reproduce
1. As owner-pro, invite `ws-viewer@doable.me` with role `viewer` to workspace `492bdda5-...`
   - Invite token: `55bacd8f6e06d34afa42cfa756a77d38056c72a1b3aa18fd6b342c162fd68ba6`
2. Log in as `owner-pro@doable.me` (a different user)
3. POST `/workspaces/invite/accept` with the token, using owner-pro's JWT
4. Observe: accept succeeds — owner-pro joins as viewer (their role in that workspace drops from owner to viewer)

## Expected
`400` or `403`: "This invite was not sent to your email address"

## Actual
`200`: invite accepted. `owner-pro@doable.me` is now listed as `viewer` in their own workspace (role demoted from `owner` to `viewer`). Cascading side effect: owner-pro can no longer perform owner-level actions (transfer, delete workspace).

## Impact
- Any user who intercepts or is forwarded an invite link can use it to join a workspace they were not invited to
- Compounded by the shareable invite link feature (which uses the same accept endpoint) — but shareable links are intentionally multi-use, so the email check should only apply to email-targeted invites
- Owner role corruption as observed side effect

## Recommended Fix
In `acceptInvite(token, userId)`: if `invite.email !== '__invite_link__'`, verify that `invite.email === callerUser.email` before proceeding.
