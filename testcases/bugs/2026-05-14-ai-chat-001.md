# BUG-AI-CHAT-001: Dev API server returns 502 Bad Gateway on all endpoints

**Date:** 2026-05-14
**Severity:** CRITICAL (blocks all AI chat testing)
**Environment:** dev (dev-api.doable.me)
**Reporter:** QA Tester Agent

## Summary
The dev API server (https://dev-api.doable.me) returns HTTP 502 Bad Gateway on ALL endpoints,
making it impossible to authenticate or use any API functionality.

## Steps to Reproduce
1. Navigate to https://dev.doable.me/login
2. Enter valid credentials (owner-pro@doable.me / TestPass123!)
3. Click "Sign in"

## Expected Behavior
- POST /auth/login returns 200 with session token
- User is redirected to dashboard

## Actual Behavior
- Browser console shows: "Access to fetch at 'https://dev-api.doable.me/auth/login' from origin
  'https://dev.doable.me' has been blocked by CORS policy: Response to preflight request doesn't
  pass access control check: No 'Access-Control-Allow-Origin' header is present"
- Direct curl/PowerShell requests: HTTP 502 Bad Gateway
- API health endpoint (https://dev-api.doable.me/health) also returns 502
- Login page gets stuck in a loop

## Affected Endpoints
- GET  https://dev-api.doable.me/ → 502
- POST https://dev-api.doable.me/auth/login → 502
- GET  https://dev-api.doable.me/auth/me → 502
- GET  https://dev-api.doable.me/chat/sessions → 502
- GET  https://dev-api.doable.me/billing/balance → 502
- GET  https://dev-api.doable.me/health → 502

## Impact
- 100% of AI chat test cases (TC-AI-CHAT-SEND, MODES, CREDITS, TOOLS, HISTORY, MODELS) cannot be executed
- Users cannot log in via email/password
- The Cloudflare tunnel or the backend API server process appears to be down

## Evidence
- PowerShell test: All endpoints return 502
- Browser console: CORS preflight failures (because 502 has no CORS headers)
- Login page: "Too many requests" error from rate limiter on earlier attempts (rate limiter is alive but API is not)

## Secondary Bug Found During Investigation

### BUG-AI-CHAT-001b: Login page pre-fills wrong email (ws-member@doable.me)
The login form pre-fills ws-member@doable.me as the email value despite the user typing
owner-pro@doable.me. React state appears to reset on re-render, restoring a cached value.
This causes silent submission failures when the form submits with the wrong email.

**Severity:** HIGH
