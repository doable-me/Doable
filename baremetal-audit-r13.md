## [BLOCKER-1] cloudflared `config.yml` ingress uses two-level subdomains
File: `/etc/cloudflared/config.yml` generated by `deployment/server-setup.sh`
Problem: Generated ingress had `hostname: api.dev.doable.me` and `hostname: ws.dev.doable.me`. Cloudflare's free Universal SSL covers only `<zone>` and `*.<zone>` (one level). The two-level names handshake-fail. Per `CLAUDE.md` the rule is `<env>-api.doable.me` / `<env>-ws.doable.me`.
Additionally: DNS for `dev.doable.me`/`dev-api`/`dev-ws` was bound to the OLD tunnel UUID (`939d91e1-...`, 0 connections) instead of the new `doable-dev-doable-me` tunnel (`fd579bd2-...`).
Repro: `curl -I https://dev.doable.me/` returns `530` (Cloudflare → origin unreachable).
Fix (applied manually in QA):
- Rewrote config to dashed hostnames (3 entries → 4000/4001/3000, catchall → 8080).
- `cloudflared tunnel route dns -f fd579bd2-bf0d-4bdf-a4bf-fa5d2de7ec30` for each of the 3 dashed hosts.
- `systemctl restart cloudflared`.
Source-level fix needed: `deployment/server-setup.sh` must (a) detect multi-label DOMAIN and emit dashed ingress, AND (b) `cloudflared tunnel route dns` for each host on FIRST tunnel creation, AND (c) re-run the route-bind when the tunnel UUID changes (it currently doesn't, so re-runs leave DNS pointed at the dead tunnel).

## [BLOCKER-2] `.env` and `apps/web/.env.local` write OAuth + NEXT_PUBLIC URLs in dotted form
Files: `/root/doable/.env` (lines ~37, 38, 44, 47-49, 53-54), `/root/doable/apps/web/.env.local`
Problem: Eight URL lines written by `setup-server.sh` use `https://api.dev.doable.me/...` and `wss://ws.dev.doable.me`. Same root cause as BLOCKER-1: the multi-label dashed-rewrite logic isn't being applied to the env-var emission step. Result: Next.js client bundle bakes the dotted URLs, every fetch from the page hits a non-resolvable host, login UI shows the generic "Something went wrong. Please try again." (apiLogin throws TypeError: Failed to fetch, hits the non-ApiError fallback in `apps/web/src/app/(auth)/login/page.tsx:137`).
Fix (applied manually in QA): `sed` rewrite + `pkill next` + `rm -rf .next .turbo` + clean restart. Open a NEW Chrome tab — stale chunks survive Ctrl+Shift+R in the original tab.
Source-level fix: audit every `.env` write path in `setup-server.sh` (and any docker .env.example) to use the dashed form. Add a `loadEnvConfig` smoke test in CI that confirms `NEXT_PUBLIC_API_URL` resolves over public DNS before declaring boot success.

## [BLOCKER-3] Sandbox EACCES cascade — preview never starts, chat shows "Connection to AI failed"
Files: `services/api/src/projects/link-sdk.ts:46`, `services/api/src/projects/dev-server-start.ts:200`, sandbox composers (`nft-egress`, `cgroup-cap`, `mac-profile`).
Problem: On every new project creation, the dev-server start path tries to write `@doable/sdk/package.json` INTO the project's per-uid sandbox dir AFTER the chown to per-project uid. EACCES.
Cascading R13-known-gap log lines:
```
[DevServer] Failed to link @doable/sdk: EACCES ... @doable/sdk/package.json
[nft-egress]  EACCES on .sandbox — skipping nft rules write (R13 known gap)
[cgroup-cap]  EACCES on .sandbox — skipping cgroup wrap (R13 known gap)
[mac-profile] apparmor_parser -r ... /var/cache/apparmor: Permission denied
[DevServer] Server for project <id> exited with code 1
```
Net effect: GET `/projects/:id/preview-url` returns 500 in a loop; the editor renders "Setting up your workspace... Downloading packages... (Ns)" forever; the chat panel shows red "Connection to AI failed" banner; **no ai_messages row is ever persisted** for the prompt.
Repro (UI): admin user → dashboard → paste any non-trivial prompt → submit → editor opens → red error. DB: `SELECT count(*) FROM ai_messages WHERE session_id IN (SELECT id FROM ai_sessions WHERE project_id=…)` → 0.
Fix candidates:
1. Link `@doable/sdk` BEFORE the per-project uid chown.
2. chown the symlinked sdk dir to the per-project uid before write.
3. Move `@doable/sdk` to a tmpfs path the doable user owns for ALL uids.

This is the headline release blocker — even with hardening relaxed, the chat-to-preview happy path is broken on a clean install.

## [MEDIUM-1] `services/api/src/projects/` and `apps/web/src/app/(dashboard)/{admin,projects}/...` files deleted on a fresh install
Files: 6 source files marked `D` in `git status` after `setup-server.sh` finishes:
- `apps/web/src/app/(dashboard)/admin/projects/page.tsx`
- `apps/web/src/app/(dashboard)/projects/[id]/settings/page.tsx`
- `services/api/src/routes/projects/api-keys.ts`
- `services/api/src/routes/projects/helpers.ts`
- `services/api/src/routes/projects/item-routes.ts`
- `services/api/src/routes/projects/list-routes.ts`
Plus the entire `services/api/src/projects/` directory was missing on first inspection.

Mechanism unknown. `setup-server.sh` references `vite-jail.ts` in a comment and uses `services/api/projects` (RUNTIME data dir, distinct from `services/api/src/projects` — source code) and `chown -R 10001-65000 services/api/projects/*` patterns. Hypothesis: a glob that intended to chown the runtime dir mis-matched the source dir, and the bare-metal hardening wiped owner-readable files. Needs an audit of every `rm`/`find -delete`/`chown -R` path in `setup-server.sh` and any sandbox-spawn helper for `*projects*` matches.

Fix: `cd /root/doable && git status -s | awk '/^ D /{print $2}' | xargs git restore`.

Note: ~893 additional files showed as `M` (modified) — almost entirely line-ending churn (`216267 insertions, 217924 deletions` for line-ending-only changes). Separate from this finding but worth a `core.autocrlf=input` check.

## [MEDIUM-2] `apparmor_parser` cannot write `/var/cache/apparmor` at runtime
Files: `services/api/src/sandbox/mac-profile.ts`, `setup-server.sh` AppArmor install step.
Problem: `[mac-profile] apparmor_parser -r /etc/apparmor.d/doable-ai-bash exited 1: Failed setting up policy cache (/var/cache/apparmor): Permission denied`. `setup-server.sh` runs `apparmor_parser -r` at install time as root (succeeds, warms cache), but the runtime `mac-profile` composer also calls `apparmor_parser -r` as the doable user — which lacks write access to `/var/cache/apparmor`.
Impact: AppArmor confinement of preview/sandbox spawns is silently bypassed at runtime. The kernel still honors the in-memory profile installed at boot, so the immediate security loss is bounded, but any policy reload from the API will fail-open.
Fix: either skip the runtime reload (don't expose `apparmor_parser -r` from non-root code paths), or pre-create a doable-writable cache subdir (`/var/cache/doable-apparmor/`) and point apparmor_parser at it.
