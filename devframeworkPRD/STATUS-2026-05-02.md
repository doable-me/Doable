## What works end-to-end now

- **Existing Vite-React projects:** behaviour byte-identical to pre-Phase-0
  (every hardcoded constant now flows from `viteReactAdapter.defaults` but
  produces the same string).
- **Creating a Next.js project:** scaffold from `nextjs-blank` template →
  project row gets `framework_id="nextjs-app"` → dev-server spawns `next dev`
  → builder runs `next build` → proxy uses Next.js reload patterns → AI
  receives the Next.js system prompt.
- **Cross-platform sandbox:** auto-selects per-OS (Psroot on Windows-with-PATH,
  systemd or bubblewrap on Linux, sandbox-exec on macOS).
- **Connector-bridge proxy endpoint:** `POST /__doable/connector-proxy/:integration/:action`
  is mounted, JWT-protected, allowlist-gated, audited. Calling it directly
  with a valid project JWT works (test via curl). Migrations 061 + 062
  applied to local dev DB; `connector_audit` and `project_runtime` tables
  exist.
- **Caddy public-site passthrough:** `*.doable.me/__doable/connector-proxy/*`
  is wired to reverse-proxy to the API. Effective on next
  `applyCaddyConfig()` regeneration.
- **Systemd per-app template:** `setup-server.sh` now installs
  `doable-app@.service` + `doable-app@.socket` + `doable-apps.target`. A
  fresh server install gets the per-app runtime infrastructure for free.

## What's still missing for production-ready Next.js apps + working static-app connectors

### Phase 5 — Pipeline integration (the load-bearing follow-up)

The Next.js adapter's `serve()` returns a spec, the runtime adapter's
`start()` writes the per-app drop-in + enables socket activation, and
caddy-admin can add the per-host route. **None of these are yet called from
the publish pipeline.**

To wire end-to-end:
1. Extend `services/api/src/deploy/adapters/doable-cloud.ts:deploy()` with:
   - look up the project's framework + runtime kind
   - if process: copy `.next/standalone/` to `/data/projects/{id}/dist-server/`
     (instead of just `/data/sites/{slug}/live/`), then call
     `nodeStandaloneAdapter.start(ctx)` + `addProcessRoute({slug, hostname,
     upstream})`, then INSERT into `project_runtime`
   - if static: existing file-copy path unchanged
2. Add an unpublish hook that calls `nodeStandaloneAdapter.stop(handle)` +
   `removeRoute(slug)` + DELETE the `project_runtime` row.

Estimated effort: ~1-2 hours of careful editing + smoke testing in a fresh
project.

### Phase 5 — Supervisor loop

PRD 06 §4.4 specifies a supervisor that subscribes to `journalctl --output=json
-f` and updates `project_runtime.state` when a unit goes failed. Without it,
crash recovery still works (systemd's Restart=on-failure handles it) but the
DB / editor UI won't reflect the state change.

Estimated effort: ~3-4 hours.

### PRD 10 — Editor UI postMessage handshake

The recent iframe-sandbox commit (`88de0b3`) made preview iframes
opaque-origin → cookies are inaccessible. A static SPA running in the
preview can't read a `doable_project_session` cookie. The fix per PRD 10
§10.6 is to issue the JWT via `postMessage` from the editor host to the
iframe on iframe load.

To wire:
1. Editor: on iframe load, fetch `POST /api/projects/:id/connector-proxy-token`
   from the host page (already authed via the user session), then
   `iframe.contentWindow.postMessage({type: "doable:connector-proxy-token",
   token}, "*")`.
2. Add `services/api/src/routes/projects.ts` endpoint that signs a fresh
   project-scoped JWT for the requesting user.
3. Inside the iframe / scaffolded app: a helper that listens for the
   postMessage, stores the token in memory, and includes it in `Authorization:
   Bearer` headers when calling `/__doable/connector-proxy/...`.

Estimated effort: ~2-3 hours (route + UI + injected runtime helper).

### PRD 10 — AI skill file for connector-fetch

## Team agent reliability — operational note

This session ran ~13 Opus team agents across the waves. **Roughly 50% of
the agents silently failed to claim their assigned tasks** despite
explicit prompts and timeouts.

Successful agents (8): db-migrator, types-builder, vite-extractor,
registry-bootstrap, harness-builder, pr-b-dev-server, pr-c-builder,
pr-d-file-manager, pr-e-install, pr-f-scaffolder, p2-prompts,
p5-foundation, prd10-foundation.

Failed-silent agents (5): pr-g-proxy, pr-h-ai-build, pr-g2-proxy,
pr-h2-ai-build, pr-i-templates. None of these wrote a file in 4-5 minutes
and the matching tasks had to be done directly.

Pattern observed: agents that succeeded all started writing files within
~60 seconds of spawn. The silent-failure agents never showed any disk
activity (verified via Monitor watching file mtimes). Sending a status-
check ping at T+60s sometimes unblocked them (p2-prompts) but mostly
didn't. Subsequent waves done directly via Edit/Write landed in seconds
with tsc clean.

**Recommendation for future sessions:** use agents for genuinely large
multi-file refactors with clear self-contained scope. Do small
mechanical surface migrations directly. Pre-arm a Monitor that watches
expected file mtimes BEFORE spawning so silent failures surface fast.
