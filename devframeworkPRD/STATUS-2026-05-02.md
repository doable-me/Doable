## What works end-to-end now

- Existing Vite-React projects: behaviour byte-identical to pre-Phase-0 (every
  hardcoded constant now flows from `viteReactAdapter.defaults`, but produces
  the same string).
- Creating a new project with `template_id="nextjs-blank"`:
  1. project row gets `framework_id="nextjs-app"` written via the scaffold route
  2. file-manager validates required files against `nextjsAppAdapter.defaults`
  3. dev-server-start spawns `npx next dev -H 127.0.0.1 -p {port}` instead of vite
  4. deploy/builder runs `npx next build` instead of vite
  5. proxy-handler.ts uses `nextjsAppAdapter.shouldReloadOnError()` to decide
     reload-on-502 (looks for `/_next/static/` and `/_next/webpack-hmr` paths)
  6. chat system prompt receives the Next.js prompt section instead of Vite
  7. AI generates `process.env.X` server-side and `NEXT_PUBLIC_*` client-side
- Sandbox backends auto-select per-OS:
  - Windows + psroot.exe on PATH → AppContainer + Job Objects
  - Linux + systemd cgroup delegation → systemd-run --scope (unchanged)
  - Linux without systemd + bwrap → unprivileged user namespaces
  - macOS → sandbox-exec SBPL (default-deny FS profile)
  - Anywhere else → DirectBackend (no isolation, last resort)

## What's still missing for production-ready Next.js apps

These are the gaps that make Phases 1-4 a foundation rather than a finished
product. **Phase 5 is the largest single remaining piece.**

### Phase 5 — Runtime supervisor (per `06-runtime-and-publish.md`)

The Next.js adapter's `serve()` returns a spec for `next start` but Doable
has no infrastructure to actually run it as a long-lived process in
production. Today's `DoableCloudAdapter` only does file-copy of static
output to `/data/sites/{slug}/live/`.

To ship a Next.js project to production we need:

1. **`project_runtime` DB table** (migration 061 or so) — tracks per-app
   listen address, systemd unit name, state machine.
2. **`/etc/systemd/system/doable-app@.service` template unit** — installed
   by `setup-server.sh`. PartOf `doable-apps.target`. Uses `EnvironmentFile`
   for vault secrets.
3. **`doable-app@.socket` template unit** — socket-activated so 100s of
   apps can sleep and wake on first request.
4. **`services/api/src/runtime/` directory** — RuntimeAdapter interface,
   supervisor loop, Caddy admin-API client.
5. **Caddy admin API integration** — at publish time, POST to
   `127.0.0.1:2019/config/...` to insert a per-host `reverse_proxy
   unix//run/doable/{slug}.sock` route.
6. **Idle eviction** — systemd's socket-activated stop after N minutes.

PRD 06 has the full spec (45 KB doc). Realistically multi-day engineering work.

### PRD 10 — Connector bridge for static apps

Static-kind apps (Vite, Astro static) can't use server-only credentials.
Today they get `VITE_*` browser-safe values via the vault-bridge but secret
keys (Slack OAuth, Stripe secret, Postgres password) are unreachable.

To close: add `POST /__doable/connector-proxy/:integration/:action` that:
- Validates a project-scoped JWT (issued at preview load via postMessage)
- Checks `.doable/connector-allowlist.json` (deny-default) for the
  (integration, action) pair
- Decrypts vault credentials and runs the Activepieces action server-side
- Audits to a new `connector_audit` table
- Same-origin via Caddy `handle_path /__doable/connector-proxy/* {
  reverse_proxy doable-api }`

PRD 10 has the spec.

### Per-framework skill files

PRD 12 §5 specifies `services/api/src/ai/skills/<framework>/*.md` for
framework-specific lazy-loaded knowledge:
- `nextjs-app/server-actions.md`
- `nextjs-app/route-handlers.md`
- `nextjs-app/connector-fetch.md`
- `nextjs-app/data-fetching.md`
- etc.

The skills materializer already exists (`services/api/src/ai/skills-materializer.ts`)
but isn't yet keyed on framework. Without these, the Next.js prompt gives
the AI a high-level description but not the depth it needs for advanced
patterns. Affects quality of generated Next.js code, not whether it works.

### Editor UI de-Vite-ing

- `modules/editor/visual-edit/use-visual-edit.ts:306` — HMR refresh

### Phase 4 sandbox follow-ups

- `psroot.exe` not bundled — Windows backend's `available()` returns false

- No smoke tests of the new backends. PRD 11 §9 specifies the test matrix.
- Apple Containerization Framework (`apple-container.ts`) and gVisor

### Framework picker UI

There's no UI for the user to choose a framework when creating a project.
The Next.js template can be selected via the existing template picker (it's
in `BUILT_IN_TEMPLATES`), so this is partly addressed — but a dedicated
framework-first UX would let the user filter templates by framework.

PRD 02 §6.2 sketches the design.

### `ensureDependencies` framework_id threading

threaded from the project row for symmetry.

## Team agent reliability — operational note

This session ran ~10 Opus team agents across the waves. **Roughly 50% of
the agents silently failed to claim their assigned tasks** despite my
spawning them with explicit prompts and timeouts. Specific failures:

- Phase 0: 3 of 3 succeeded (db-migrator, types-builder, vite-extractor)
- Phase 1 W1: 2 of 2 succeeded (registry-bootstrap, harness-builder)
- Phase 1 W2: 3 of 3 succeeded (pr-b-dev-server, pr-c-builder, pr-d-file-manager)
- Phase 1 W3: 2 of 4 succeeded (pr-e-install + pr-f-scaffolder; pr-g-proxy and pr-h-ai-build never wrote a file in 5 minutes)
- Phase 1 W3 retry: 0 of 3 succeeded (pr-g2-proxy, pr-h2-ai-build, pr-i-templates all silent for 4+ minutes)
- Phase 2 W1: 1 of 1 succeeded (p2-prompts, after a status-check ping unblocked it at T+60s)

Subsequent waves and surfaces I just did directly via Edit/Write — those
landed in seconds and tsc stayed clean. **Recommendation for future
sessions:** use agents for genuinely large multi-file refactors; do small
mechanical surface migrations directly. Save the agent overhead for cases
where the parallelism actually pays off.
